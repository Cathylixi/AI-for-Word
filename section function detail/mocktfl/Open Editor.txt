Open Editor

Overview
This document explains the current MockTFL "Open Editor" logic in AI-for-Word. The purpose of this feature is to let the user open a dedicated dialog for one specific MockTFL work item, restore any previously saved section data if it exists, optionally run the MockTFL analysis workflow, review and edit the generated Notes, choose dynamic table columns, insert the result back into Word, and save the section snapshot into the MockTFL database state.

This part of the system is not just a popup editor. It is the orchestration layer that connects:
- the selected MockTFL item in the taskpane
- the title/body content controls already inserted into Word
- the MockTFL dialog UI
- backend SAP and CRF analysis APIs
- Word table insertion logic
- MockTFL section-level database persistence

End-to-End Purpose
The feature solves the following problem:
- After the user inserts the MockTFL template, they need to work on one specific table/figure/listing at a time.
- The system must know exactly which Word location belongs to that work item.
- The system must be able to reopen a previously saved section and restore all related analysis text.
- For supported prototype items, the system must guide the user through SAP matching, variable extraction, CRF mapping, and final insertion.
- For supported tables, the system must insert a structured Word table instead of plain text.

High-Level Flow
The current flow is:
1. User selects a MockTFL work item in the "Start working on" dropdown.
2. Frontend parses the selected value and selects the corresponding body content control in Word.
3. User clicks "Open Editor".
4. Frontend opens a separate Office dialog for MockTFL editing.
5. The dialog tells the parent page it is ready.
6. The parent reads the Word title content control for the selected work item.
7. The parent loads the current MockTFL state from the database.
8. The parent checks whether the current section already has saved data.
9. If saved data exists, the parent sends that saved data into the dialog.
10. If no saved data exists, the parent sends an empty initialization message into the dialog.
11. The dialog renders the current item title, dynamic column options, and any saved analysis content.
12. For supported items, the user can click "Start analysis".
13. The parent sends the selected item title to the backend for SAP matching and variable extraction.
14. The user reviews or edits the suggested variables.
15. The user clicks "Confirm Variables & Map to CRF".
16. The parent sends the confirmed variables to the backend for CRF mapping.
17. The backend returns grouped CRF mapping results, which are converted into final Notes text.
18. The user may edit the Notes text and choose table columns.
19. The user clicks "Insert".
20. The parent inserts the result back into the correct Word body content control.
21. The parent saves the section snapshot into the MockTFL database state.

Detailed Logic

Step 1: User selects a MockTFL work item in the taskpane
Files:
- frontend/src/taskpane/controller/appController.js
- frontend/src/taskpane/domains/mocktfl/controller/mockTflWorkController.js

Behavior:
- The selected value from the dropdown has the format:
  `TYPE:NUMBER`
  Example:
  `TABLE:14.1.1.2`
- When the user changes the dropdown, the taskpane calls:
  `handleMockTflWorkItemChange({ studyNumber, value })`

Inside `handleMockTflWorkItemChange(...)`:
- It calls `parseMockTflWorkItemValue(value)` to split the string into:
  - `type`
  - `number`
- It then calls:
  `selectMockTflBody({ studyNumber, type, number })`

Purpose:
- To identify exactly which MockTFL item the user wants to work on.
- To move Word selection to the correct body placeholder before the dialog opens.

Effect:
- The corresponding MockTFL body content control in Word becomes selected.
- The `Open Editor` button becomes available.

Step 2: Select the target body content control in Word
File:
- frontend/src/taskpane/domains/mocktfl/word/sections.js

Function:
- `selectMockTflBody({ studyNumber, type, number })`

Behavior:
- Uses `buildMockTflBodyTag(...)` to generate the body content control tag.
- Runs `Word.run(...)`.
- Gets the content control by tag using:
  `context.document.contentControls.getByTag(tag)`
- Selects the first matched content control.

Used APIs:
- Office Word JavaScript API
- `Word.run`
- `contentControls.getByTag`
- `select`

Purpose:
- To anchor the current work item to the correct Word location.

Effect:
- Later insertion logic already knows the correct placeholder region in the document.

Step 3: User clicks Open Editor
Files:
- frontend/src/taskpane/controller/appController.js
- frontend/src/taskpane/domains/mocktfl/controller/mockTflWorkController.js

Behavior in `appController.js`:
- Reads the selected work item from `sectionSelect`.
- Calls `parseMockTflWorkItemValue(selection)`.
- Looks up the entry in `insertedMockTflEntries`.
- Builds a readable label using:
  `buildMockTflLabel({ type, number, title })`
- Calls:
  `openMockTflEditorDialogForWorkItem({ studyNumber, type, number, label })`

Purpose:
- To open a dedicated editor for the current MockTFL item, with enough context to restore or analyze it.

Effect:
- The separate MockTFL dialog workflow starts.

Step 4: Open the MockTFL editor dialog
File:
- frontend/src/taskpane/domains/mocktfl/controller/mockTflWorkController.js

Function:
- `openMockTflEditorDialogForWorkItem({ studyNumber, type, number, label })`

Behavior:
- Checks whether the Office dialog API exists:
  `Office.context.ui.displayDialogAsync`
- Closes any previous dialog with:
  `closeMockTflEditorDialog()`
- Builds the URL:
  `window.location.origin + "/mockTflEditorDialog.html"`
- Opens the dialog via:
  `Office.context.ui.displayDialogAsync(dialogUrl, { height: 70, width: 40 }, ...)`

Used APIs:
- Office Dialog API
- `Office.context.ui.displayDialogAsync`

Purpose:
- To launch a dedicated workspace for one MockTFL item.

Effect:
- A new dialog window opens for the selected MockTFL item.

Step 5: Register dialog event handlers
File:
- frontend/src/taskpane/domains/mocktfl/controller/mockTflWorkController.js

  - `mocktfl:clear`
  - `mocktfl:close`

Purpose:

Effect:
- The parent becomes the orchestrator for all later dialog actions.

Step 7: Parent reads the Word title content control
Files:
- frontend/src/taskpane/domains/mocktfl/controller/mockTflWorkController.js
- frontend/src/taskpane/domains/mocktfl/word/sections.js

Behavior after receiving `mocktfl:dialogReady`:
- Parent calls:
  `readMockTflTitle({ studyNumber, type, number })`
- This reads the title content control text from Word.
- Parent then calls:
  `extractPureTitleFromMockTflTitleCcText(titleCcText)`

Purpose:
- To use the actual Word title content control as the source of truth.
- To derive a semantic title without the `TABLE 14.x` / `FIGURE ...` prefix.

Effect:
- Parent obtains:
  - `titleCcText`
  - `pureTitle`

Why this matters:
- If the user edits the title directly in Word, analysis still uses the updated document title.
- The pure title is later used for embedding and SAP/CRF matching.

Step 18: Backend Step 1 analysis service
File:
- backend/services/mocktfl/mockTflStartAnalysisService.js

Function:
- `startMockTflAnalysis({ studyNumber, pureTitle, mongoose, openaiApiKey })`

Main logic:
1. Create an embedding for `pureTitle`
2. Match that embedding against saved SAP section title embeddings
3. Read the matched SAP section text
4. Reuse saved SAP metadata such as `matchedCrfFormName`
5. Send the matched SAP section text to GPT
6. Ask GPT to return the statistical variables needed for this MockTFL item

Used APIs:
- OpenAI Embeddings API
- OpenAI Chat Completions API
- MongoDB via `mongoose.connection.client`

Databases involved:
- `llxdocument.studies`
- saved SAP state (`chunkType = "SAP"`)

Purpose:
- To determine which SAP section is most relevant and extract editable statistical variables from it.

Effect:
- Backend returns:
  - matched SAP label
  - matched SAP section text
  - suggested statistical variables
  - saved CRF form context

Step 23: Backend Step 2 mapping service
File:
- backend/services/mocktfl/mockTflStartAnalysisService.js

Function:
- `mapMockTflVariablesToCrf({ studyNumber, pureTitle, variables, variablesText, savedCrfFormName, mongoose, openaiApiKey })`

Main logic:
1. Normalize the confirmed variables
2. Recreate the embedding for `pureTitle`
3. Resolve which CRF form should be used
4. Prefer the CRF form name saved during SAP generation
5. Fallback to CRF title-embedding match if necessary
6. Match each confirmed variable to the best CRF question
7. Gather the corresponding CRF answers/options
8. Ask GPT to rewrite matched CRF questions into formal noun-phrase style table row headers

Databases involved:
- `llxdocument.studies`
- CRF chunks (`chunkType = "crf_crfFormList"`)

Used APIs:
- OpenAI Embeddings API
- OpenAI Chat Completions API
- MongoDB via `mongoose.connection.client`

Purpose:
- To bridge the selected MockTFL item back to structured CRF data, while turning raw form questions into cleaner statistical row labels.

Effect:
- Backend returns:
  - matched CRF form info
  - question list
  - detailed variable-to-question mappings

Step 24: Parent converts raw CRF mappings into UI-ready text
File:
- frontend/src/taskpane/domains/mocktfl/controller/mockTflWorkController.js

- `groupCrfMappingsByQuestion(mappings)`
- `formatGroupedMappingsForEvidence(groupedMappings)`
- `formatGroupedMappingsForNotes(groupedMappings)`

Behavior:

Purpose:

Effect:

Step 28: Structured table insertion for supported MockTFL tables
Files:
- frontend/src/taskpane/domains/mocktfl/controller/mockTflWorkController.js
- frontend/src/taskpane/domains/mocktfl/word/tables/table14_1_1_2_patientDisposition.js

Current supported table insert targets:
- `TABLE 14.1.1.2`
- `TABLE 14.1.3.1`

Behavior:
- If the item is a supported structured table target:
  - For `14.1.1.2`, call:
    `insertTable14_1_1_2_PatientDisposition({ studyNumber, text, selectedColumns })`
  - For `14.1.3.1`, call:
    `insertTable14_1_3_1_SummaryOfDemographics({ studyNumber, text, selectedColumns })`

Inside these insertion functions:
- Read line 3 of the title content control using:
  `readMockTflTitleLine3(...)`
- Build the target body content control tag
- Find the target content control in Word
- Parse the Notes text into table rows
- Build a fixed-layout OOXML table
- Clear the placeholder content control
- Insert the OOXML table inside the content control
- Append footnotes after the table

Used APIs:
- Office Word JavaScript API
- `Word.run`
- `contentControls.getByTag`
- `clear`
- `insertOoxml`
- `insertParagraph`

Purpose:
- To replace the placeholder with a real Word table that follows MockTFL formatting rules.

Effect:
- The Word document shows a structured table with dynamic columns and footnotes.

Step 34: Error handling across the dialog workflow
Files:
- frontend/src/taskpane/domains/mocktfl/controller/mockTflWorkController.js
- frontend/src/taskpane/dialog/mocktfl/mockTflEditorDialog.js

Behavior:
- The parent wraps each dialog-driven action in `try/catch`.
- If any step fails, the parent sends:
  `mocktfl:error`
- The dialog then:
  - restores button states if possible
  - displays the error message in status

Purpose:
- To prevent the dialog from getting stuck in a disabled state when backend or Word operations fail.

Effect:
- The user sees a recoverable error message rather than a silent failure.

What Open Editor Does Not Do
The current Open Editor workflow does not:
- support the full analysis prototype for all MockTFL items
- guarantee a structured Word insertion path for all non-allowlisted items
- automatically run analysis when the dialog opens
- skip database loading; it always checks saved state first

Current Limitations
There are several implementation limitations in the current version:
- Full analysis is currently gated to two tables only.
- Structured table insertion is currently gated to the same two tables only.
- Non-allowlisted items do not currently have an equally complete insert path inside this dialog controller.
- The workflow depends on the earlier Insert Template step having already created the correct title/body content controls in Word.

Summary
The current MockTFL Open Editor logic is a dialog-based orchestration workflow.

Its exact role is:
- identify the selected MockTFL work item
- open the item-specific dialog
- read the real Word title content control as the source of truth
- load saved section data from the database if it exists
- load dynamic column options from global MockTFL state
- optionally run Step 1 SAP matching and variable extraction
- optionally run Step 2 CRF mapping
- present evidence and final Notes to the user
- insert structured output back into the correct Word body content control
- save the finished section snapshot into the MockTFL database

Most importantly, Open Editor is the bridge that connects the inserted Word placeholders, the AI analysis workflow, and the section-level persistence model.
